# Lessons Learned — 真实翻车点登记册 & 维护流程

Companion to `SKILL.md`. 这是 `trip-planner` 的「活文档」记忆：把每次真实运行里踩过的坑固化成规则，
让同一个错误不再发生第二次。每条都来自实际 session，不是假想的需求。**SKILL.md 只保留蒸馏后的规则；
完整的「症状 → 根因 → 现在的规则」叙事放这里**，这样主文件保持精简，而经验不丢。

## 怎么用这个文件

- **执行行程时**：不用通读。SKILL.md 的规则就是这些教训的结论；只有当你想知道「为什么这条规则这么严」
  时，回这里看对应的 L 条。
- **维护 skill 时**：每次真实运行后，按文末「维护流程」把新坑登记成一条新的 L，并把蒸馏出的规则写进
  SKILL.md（必要时加一条 eval、加一条 `check_html.py` 校验）。

每条登记格式固定：
**症状**（真实运行里发生了什么）→ **根因** → **固化规则**（现在写在 SKILL.md 哪里）
→ **关联校验**（`check_html.py` 第几条，如有）→ **记录日期**。

## 翻车点登记册

### L1 · 纯火车行程没连浏览器，酒店直接走估价

- **症状**：一趟纯高铁出行里，agent 以为「只有机票才是 skill 强制要浏览器的那一项」，于是压根没去连
  Chrome，酒店的价格和评论都直接走了 `web_search` 估价。
- **根因**：把「浏览器强制」错误地和「这趟有没有机票」绑定了。
- **固化规则**：SKILL.md Step 5 ——「选酒店（非已订）时，价格与评论都必须**浏览器实查**，纪律与机票同级；
  有没有机票，与酒店要不要连浏览器，毫无关系」。上位规则见 数据诚信契约「浏览器优先，不是浏览器可选」。
- **关联校验**：check #10（酒店价格第二信源）部分兜底。
- **记录日期**：2026-06。

### L2 · 把「评论双信源」当成「查两家就完事」，丢了飞猪价格

- **症状**：一次运行里 agent 把「评论双信源 = 携程 + 高德」理解成「查两家就完事」，于是只碰了携程和高德，
  把飞猪的**价格**比价整个丢了。
- **根因**：混淆了两套**互相独立**的多源要求——价格多源 ≠ 评论多源。
- **固化规则**：SKILL.md Step 5 的「价格 vs 评论」双表——价格比价必查 **携程 + 飞猪**（国际加 Booking/Agoda），
  评论核验必查 **携程 + 高德**；一张合格酒店卡 = 携程（价+评）+ 飞猪（价）+ 高德（评）三次浏览器读。
- **关联校验**：check #10（携程-only 价格 = 单源，FAIL）。
- **记录日期**：2026-06。

### L3 · 一个源失败，连累另一个源一起放弃

- **症状**：一次运行里高德评论没加载出来，agent 就写「高德/飞猪我没核到」——把价格源（飞猪）和评论源
  （高德）捆在一起一起放弃了。
- **根因**：把不同维度、不同平台的失败错误地传染成「这家我整体没法查」。
- **固化规则**：SKILL.md Step 5 ——「一个源失败，绝不连累另一个」。高德（评论源）失败，与飞猪（价格源）该不该
  查毫无关系；反之亦然。一个挂了就单独如实标它挂了，另一个照查。
- **关联校验**：—（纪律项，无专门校验）。
- **记录日期**：2026-06。

### L4 · 高德评论懒加载，太早下「没评价」结论

- **症状**：点开高德 POI 后立刻读评价，读到 0 条就下「这家没评价」的结论；或搜索默认跳到北京（没解析到
  目标城市）也以为是「没数据」。
- **根因**：高德评价是懒加载，且 query 没带城市名时会错误定位——「取到 0 条」十有八九是没等够 / 没定位对，
  不是真没有。
- **固化规则**：SKILL.md Step 5 ——「点开 POI 后 wait 3–5s、必要时在详情面板内下滚一下再读；搜索若跳了
  北京就把城市名加进 query 重搜+点搜索按钮」。抓取自愈细节见 `scraping-method.md` §八。
- **关联校验**：—（方法项）。
- **记录日期**：2026-06。

### L5 · 编造酒店评分 / 点评数 / 评论结论

- **症状**：没真读评论区，却写出「4.8 分 · 3,201 条点评（已核）」这类看起来可信的具体值/结论。
- **根因**：默认路径会去「填满」评分与评论字段，而不是承认没查到。
- **固化规则**：数据诚信契约「受控『已验证』词表——只有真查到才能用」+ 分项硬规矩表（没查到只能写
  「评分以 App 为准」，禁止任何分数/条数）。
- **关联校验**：check #8（评论结论/评分必须带 `实查于 YYYY-MM-DD` 同卡来源，否则 FAIL）。
- **记录日期**：2026-06。

### L6 · 多日复制同一温度串，当成各日预报

- **症状**：对 14 天外的行程，把同一个具体温度（如「约 28°/19°」）复制到每一天的卡片，伪装成逐日预报。
- **根因**：14 天外根本拿不到逐日预报，但模板留了温度位，于是被填上同一个数。
- **固化规则**：数据诚信契约分项表「天气（>14 天外）」——只写「X月气候典型 约A°/B°（非当日预报，出行前再查）」，
  不得对多日复制同一具体温度串当各日预报。
- **关联校验**：check #11（同一温度串重复出现且无「气候典型/非当日预报」限定 = FAIL）。
- **记录日期**：2026-06。

### L7 · 假「✓最低」与已废平台当价格源

- **症状**：给一行本身是「（估）」的价格打了「✓最低」标；或把去哪儿/美团/同程/艺龙等已废平台当价格源放进比价。
- **根因**：没有真实比过价就钦定「最低」；沿用了已经停止运营/不可信的平台清单。
- **固化规则**：数据诚信契约分项表「酒店/机票比价」——估价行不得打「✓最低」；已废平台（去哪儿/美团/同程/艺龙）
  禁止出现。现役平台真值表见 `research-playbook.md`。
- **关联校验**：check #9（估价行带 .cheapest，或已废平台当价格源 = FAIL）。
- **记录日期**：2026-06。

## 对标观察登记册（生态对标 → 取舍）

L 系列记「自己踩的坑」；这一节记「评估过的同类/外部工具，以及为什么纳入或不纳入」。
认真评估过一个外部工具，无论结论是用还是不用，都留一条——免得下次从零再评一遍，
也免得忘了当初为什么否掉。每条格式：
**对标对象** → **评估结论** → **决定性理由** → **未来复评的唯一窄口子 + 门槛** → **复评触发** → **记录日期**。

### B1 · anysearch-skill（搜索 API/skill）——不纳入

- **对标对象**：[anysearch-skill](https://github.com/anysearch-ai/anysearch-skill)，给 AI agent 用的统一搜索
  API/skill（search 返回摘要、extract 返回整页 markdown、batch 并行搜、若干垂直域）。评估于 2026-06-20，
  以下事实为当时实查：2026-05-11 香港发布；免费档 1000 次/天、20 QPS；查询与 API key 都发往
  `api.anysearch.com`（无本地模式）。
- **评估结论**：**不纳入工作流。** 它是被严格支配（strictly dominated）的添加——能插的位置，要么已有同级/更高
  可信度的工具覆盖（web_search + 已登录浏览器），要么正是数据诚信契约禁止用搜索 API 回答的地方。
- **决定性理由**（任一独立成立）：
  1. **只能改进已被降级的那一层，碰不到价值层。** 它是搜索 API（号称 real-time，但 live/cached 未定义、
     单条结果 schema 未公开），在契约里与 `web_search` 同属 `实查` 子类，对价格/评论被硬降级到浏览器之下。
     三类强制浏览器的数据（机票实时含税价、酒店实时房价、真实评论区）它一类都升不进已核验层——天花板焊死。
  2. **中国主战场未被验证。** 研究骨架是携程/飞猪/高德/12306/小红书；anysearch 有 Travel 垂直域的*宣传*，
     但全网零中国平台被点名，定位英文/企业数据并明说「刻意不索引开放消费级网络」——而中国 OTA/地图/铁路/
     小红书恰恰就是开放消费级网络。能让它在此立足的差异点，正是它没有证据拥有的东西。
  3. **供应商太新、不可问责。** 发布约 6 周、创始人匿名、无融资披露；3.5k star vs ~38 commit 像发布期刷量；
     付费档无价无配额，依赖的免费档随时可改价/关停。把它焊进一个以「数字可信」为卖点的 skill 是自拆台脚。
  - 附带风险：合成式输出读起来像结论（「评价不错 / 口碑稳」、自信的远期气温），会诱发 **L5** 那类编造——
    把受控『已验证』词或 ✓最低 写进没浏览器实读的卡片。
- **未来复评的唯一窄口子 + 门槛**：若某天要重评，**只放行一个位置**——燃油附加费最新标准 / 单个官方静态页
  （营业时间、闭馆日、票价、预约页）的 `extract`，因为这些是吃新鲜度的静态官方事实、不沾三类强制浏览器数据。
  且必须**全部满足**才放行：
  1. **燃油附加费新鲜度测试通过**：返回最新标准 + 执行日期、与官方一致，不能把旧公告当现行（否则比 hedge 还糟）。
  2. **中国静态事实命中率**对 web_search 头对头有**可量化提升**，且取值正确、非合成。
  3. 只注册**一个**受控标记 `anysearch · 未浏览器核实`，继承 web_search 的全部降级（永不 ✓最低、永不带评论结论、
     远期天气仍写「气候典型 / 非当日预报」），守住「一个概念只用一个词」。
  4. `check_html.py` #8–#10 仍把它当**非浏览器源**，不能被它满足。
  5. 带 flag、web_search 仍是默认与兜底、缺它能优雅降级；**没有任何核心步骤依赖它**。
  6. 含个人行程上下文（人群 / 健康 / 具体日期）的 query **不走** anysearch。
  - **任一不过，维持不纳入。绝不让它靠近一个价格、一张机票、一条评论。**
- **复评触发**：anysearch（或同类搜索 API）公开了中国 OTA/地图/铁路的**具名覆盖 + 可验证命中率**，
  或它进入 harness 成为内置工具时，才值得按上面六门重跑一次；否则不必复评。
- **记录日期**：2026-06-20。

## 维护流程（观察 → 提炼 → 测试）

skill 是活文档，靠真实运行喂养，而不是一次写死。沿用 Anthropic「Complete Guide / 最佳实践」里的
**观察 → 提炼 → 测试**循环（即用一个 Claude 帮你改 skill、另一个全新 Claude 跑真实任务暴露问题）：

1. **真实运行 + 观察**：拿真任务（不是测试场景）跑这个 skill，盯住它在哪里挣扎、跳步、或做了意外选择。
2. **登记新坑**：在上面的「翻车点登记册」加一条新的 L（症状 → 根因），写清是哪种真实失败。
3. **提炼成规则**：把这条坑蒸馏成一句**可执行**的规则，放进 SKILL.md 最相关的位置；用「必须/绝不」这类
   强约束词，别用「尽量」。规则要短——叙事留在本文件，主文件只留结论。
4. **加测试**：在 `evals/evals.json` 加一条能复现这个坑的用例（含期望行为）；如果这个坑是**结构可检测**的
   （某字段缺来源、某平台没查、某串重复），就在 `scripts/check_html.py` 加一条对应校验，并回填本条的
   「关联校验」。
5. **复测**：用全新会话加载更新后的 skill，跑相同/相似任务，确认坑不再复现，再固化。

> 经验法则（来自官方指南）：**先把一个最难的真实任务死磕到 Claude 做对，再把「赢的做法」抽成规则**——
> 比凭空想象需求更有效。`check_html.py` 只是结构兜底，挡不住刻意绕过；真正的防线永远是写清楚的规则 +
> 「宁可不写，绝不编」。
